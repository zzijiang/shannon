# Design Lessons · 设计经验沉淀

边做边记录。每条经验注明发现场景和原因，方便后续判断是否仍然适用。

## 原型 v1：matplotlib 入门单元（2026-04-12）

### 交互设计

**1. 讲图必须有图**
- 场景：探索第 3 页说"x 轴上是 0,1,2,3,4"，但页面上没有图
- 问题：学生不知道在说什么，文字描述无法替代视觉
- 原则：**凡是讨论图表的页面，必须展示对应的图表。文字是辅助，图是主体。**

**2. 选项图表必须有刻度**
- 场景：探索第 2 页让学生猜代码输出，四个小图没有 y 轴数值
- 问题：学生无法对照 `[35,33,29,31,36]` 判断哪个图对
- 原则：**选项中的图表至少要有关键轴的刻度，让学生有判断依据。**

**3. 交互完成后必须保留结果**
- 场景：探索第 4 页点放 5 个数据点，放完后页面重绘，所有点消失
- 原因：`render()` 重建了 DOM，但交互状态（placed 数组）丢失
- 原则：**重绘时检查"已完成"状态，渲染完成态而非空白态。交互结果是学生的成果，不能吞掉。**

**4. 别考英语单词，要考编程概念**
- 场景：探索第 6 页让学生填 `bar`，但初中生可能不知道 bar = 柱子
- 问题：卡点是英语词汇，不是编程理解。学习目标错位。
- 修复：改为三选一（`plt.line` / `plt.bar` / `plt.column`），答对后解释 bar 的含义
- 原则：**填空题只用于学生已经见过的内容的回忆。首次接触新术语用选择题，降低无关门槛。**

### 代码编辑与执行

**5. Starter code 里的注释不能是答案**
- 场景：Step 5 注释写了 `# 提示：colors = ['red' if h > 3 else 'green' for h in hours]`
- 问题：学生直接复制粘贴，零学习效果
- 修复：注释只描述意图（"超过3小时 → 红色"），具体写法通过 💡 提示按钮分级提供
- 原则：**编辑器里的注释只说"做什么"，不说"怎么做"。答案放在提示系统里，让学生主动选择要多少帮助。**

**6. 错误信息必须翻译成大白话**
- 场景：学生写 `color=blue`（没加引号），Python 报 `NameError: name 'blue' is not defined`
- 问题：初中生看不懂英文错误信息，通用提示"代码有错误"也无法帮助定位
- 修复：根据错误类型翻译——NameError + 颜色词 → "忘了加引号？颜色名要写成 `'blue'`"
- 原则：**每种常见错误类型都要有对应的中文解释，指向具体的修复动作，不是笼统的"检查一下"。**

**7. "无图表"提示要和当前步骤匹配**
- 场景：Step 4 代码能运行但没画图，提示说"是不是少了 plt.plot()"
- 问题：这一步根本不需要 plt.plot()，需要的是 plt.axhline()
- 修复：每个步骤有独立的 noPlotHint，指向该步骤实际需要写的代码
- 原则：**反馈信息必须和当前上下文相关。通用提示 = 无用提示。**

### 产品机制

**8. "点击运行"不等于"学会了"**
- 场景：项目 v1 每步都是预填好的完整代码，学生只点"运行"
- 问题：看到图表有感觉，但完全无法独立写出代码。被动观看 ≠ 主动学习。
- 修复：关键步骤改为 challenge 模式，删掉核心代码，学生必须自己写
- 原则：**每个步骤至少有一处需要学生写出（非复制）的代码。代码编辑器的默认状态应该是"不完整的"，而非"已完成的"。**

**9. 代码必须真正执行**
- 场景：v1 用预录输出，学生改了代码但结果不变
- 问题：破坏了"实验→反馈"循环，学生失去探索动力
- 修复：集成 Pyodide，代码真实运行，matplotlib 真实渲染
- 原则：**编程教学产品的代码执行必须是真实的。模拟输出只能用于非编程的交互环节（如探索模式的选择题）。**
